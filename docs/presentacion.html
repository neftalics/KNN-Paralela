Introducción: Reto y Solución

1. ¿Qué es KNN?

Algoritmo supervisado que clasifica datos según la mayoría de votos de sus vecinos cercanos.


2. El Problema (Cuello de Botella)

Calcular distancias contra todo el dataset es costoso: O N × M .


3. La Solución Propuesta

Paralelizar el cálculo usando MPI y el modelo SPMD para dividir la carga entre múltiples procesadores.

Metodología (Topología DAG)

Arquitectura Maestro-Trabajador

La paralelización se modela como un grafo de tareas dirigido:

  • Inicio (Main): El nodo maestro divide el dataset (Scatter).
  • Paralelo (T1-T4): Cada trabajador calcula distancias en su fragmento de datos localmente.
  • Fin (End): Se recolectan los resultados parciales (Gather) para la decisión final.

Metodología: Flujo de Ejecución

Beta 1: Scatter

El nodo maestro lee los datos y utiliza MPI_Scatter para distribuir fragmentos equitativos del dataset de prueba a todos los nodos trabajadores.

Beta 2: Broadcast

Se utiliza MPI_Bcast para enviar el dataset de entrenamiento completo a todos los nodos. Cada nodo calcula distancias locales.

Versión Final: Gather

El maestro recolecta todas las predicciones parciales usando MPI_Gather y calcula la métrica de precisión global.

Topología del Clúster

La arquitectura implementada simula un entorno de memoria distribuida.

  • Nodo 0 (Master): Orquestador de I/O y recolección.
  • Nodos 1-N (Workers): Motores de cálculo puro.
  • Comunicación: Minimización de latencia mediante envío de bloques contiguos.

Rendimiento: Tiempo de Ejecución

  • Reducción drástica: El tiempo baja de 5s a casi 2s al pasar de 1 a 4 procesos.
  • Punto de Saturación: Observa el "codo" en 4 procesos. Añadir más nodos (8) apenas mejora el tiempo.
  • Causa: El tiempo de comunicación (línea naranja) empieza a pesar más que el ahorro en cálculo.

Análisis de Aceleración (Speedup)

  • Ideal vs Real: La línea punteada es el ideal. Nuestra curva real (azul) se aleja al aumentar N.
  • Speedup Máximo: Logramos acelerar 2.5x veces el proceso con 8 nodos.
  • Ley de Amdahl: La parte secuencial del código (I/O y envíos) limita la velocidad máxima teórica.

Eficiencia del Clúster

  • Caída de Eficiencia: Del 94% (excelente) con 2 nodos, bajamos al 31% con 8 nodos.
  • ¿Qué significa? Con 8 nodos, estamos desperdiciando el 70% de la capacidad de CPU en esperas.
  • Conclusión: Para este tamaño de dataset, 4 nodos es la configuración más rentable (60% eficiencia).

Rendimiento (GFLOPs)

  • Potencia Bruta: A pesar de la ineficiencia, la cantidad total de operaciones por segundo sube.
  • Escalabilidad Fuerte: El sistema es capaz de entregar más cómputo si se le exige más.
  • Utilidad: Esto justifica el uso del clúster si el dataset creciera de 8k a 1 millón de datos.

Escalabilidad del Problema (Datos)

  • Configuración: Se mantuvo fijo el clúster en 4 procesadores (p=4) y se varió el tamaño del dataset (N).
  • Crecimiento Cuadrático: Al duplicar N (de 4k a 8k), el tiempo se triplica (18s a 67s), confirmando la complejidad O(N^2).
  • Límite: El sistema puede manejar datasets más grandes, pero el tiempo crecerá exponencialmente.

Conclusiones del Proyecto

Paralelización Exitosa: Reducción del tiempo de 5.02s a 2.01s. El algoritmo es correcto y más rápido.

Punto Óptimo: Con 4 procesos se obtiene el mejor balance entre Speedup (2.4x) y Eficiencia (60%).

Cuello de Botella: Con 8 procesos, la ganancia es marginal debido al costo de comunicación, y con grandes datasets el tiempo crece cuadráticamente.
